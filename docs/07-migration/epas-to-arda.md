# EPAS to Arda Migration Strategy

## 📋 Overview

Migration từ EPAS (Kiến trúc cũ) sang Arda (Microservices) không chỉ là copy code mà là một quá trình **Refactor & Re-architect**.

## 🏗️ Migration Framework

### 1. Phân tích Domain (Domain Discovery)
- Xác định Table/API trong EPAS thuộc về Bounded Context nào trong Arda (Accounting, Loan, CRM...).
- Mapping từ Monolith (EPAS) sang Microservice tương ứng.

### 2. Schema Mapping (PostgreSQL)
- **EPAS**: Thường sử dụng CamelCase hoặc PascalCase cho column, thiếu RLS.
- **Arda**: Sử dụng `snake_case`, bắt buộc có `tenant_id` và kích hoạt **Row-Level Security (RLS)**.

### 3. API Mapping (REST → gRPC)
- Chuyển đổi các REST endpoint cũ sang Protobuf definitions.
- Xử lý xác thực: Từ Session-based (EPAS) sang JWT/OIDC (Zitadel).

## 🛡️ Migration Steps

### Bước 1: Mapping Database
| EPAS Table | Arda Service | Arda Table | Ghi chú |
|------------|--------------|------------|---------|
| `tblAccount` | `accounting` | `accounts` | Thêm `tenant_id` |
| `tblMember` | `crm` | `members` | Tách sang CRM service |
| `tblLoan` | `loan` | `loans` | Chuyển sang CBS Java |

### Bước 2: Viết Glue Code / Anti-Corruption Layer
- Sử dụng **Saga Orchestrator** (Camunda) để đồng bộ dữ liệu giữa hệ thống cũ và mới trong giai đoạn chuyển tiếp.
- Tạo các gRPC client trong Arda để gọi ngược lại các legacy API nếu cần.

### Bước 3: Data Migration
- Sử dụng SQL Scripts (pg_dump/copy) kết hợp với các script transform (Python/Go) để migrate dữ liệu.
- Validate tính toàn vẹn (Checksum, Count).

## 🚀 Chiến lược triển khai (Rollout)

1. **Strangler Fig Pattern**: Chuyển từng module một. Module nào quan trọng (Core Banking) chuyển sau, module vệ tinh (CRM, Notification) chuyển trước.
2. **Shadow Traffic**: Chạy song song hệ thống mới và cũ, so sánh kết quả trả về trước khi switch hoàn toàn.
